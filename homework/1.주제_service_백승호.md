# 1. 장기 프로젝트 섀도잉 코파일럿 (Long-term Project Shadowing Copilot)

## 📌 개요
특정 프로젝트(예: 신규 프레임워크 개발, 대규모 리팩토링) 시작 시 생성되어, 프로젝트 기간 내내 팀원 전체가 공용으로 활용하는 AI 어시스턴트. 프로젝트의 모든 결정 사항, 회의록, 이슈 티켓 등을 장기 기억(Long-term Memory)으로 가져가며 맥락을 유지합니다.

## 👥 Stakeholders (이해관계자)

| 종류 | 역할 | 중요하게 생각하는 품질 속성 (Quality Attributes) |
| :--- | :--- | :--- |
| **Team Member (팀원)** | 실제 프로젝트를 수행하며 코파일럿과 질의응답을 진행하고 과거 결정 사항을 확인하는 실무자. | **정확성(Accuracy)**, **사용성(Usability)**, **응답성(Performance)** |
| **Project Lead (PL / 리더)** | 프로젝트의 방향성을 관리하고, 코파일럿을 통해 산재된 문서와 대화 내용을 요약/모니터링하는 관리자. | **추적성(Traceability)**, **신뢰성(Reliability)** |
| **System Admin (관리자)** | 프로젝트 시작 시 코파일럿 워크스페이스를 세팅하고, 사내 데이터 연동 파이프라인을 관리하는 담당자. | **유지보수성(Maintainability)**, **상호운용성(Interoperability)** |

---

## 📝 Requirements (요구사항)

### Functional Requirements (기능 요구사항, FR)
* **FR-1 [프로젝트 워크스페이스 생성]:** 관리자 또는 PL은 신규 프로젝트용 코파일럿 공용 세션을 생성하고 초기 컨텍스트(목표, 참여자 등)를 주입할 수 있어야 한다. *(Source: Project Lead, System Admin)*
* **FR-2 [멀티 세션 히스토리 통합]:** 시스템은 프로젝트 팀원들이 해당 공용 세션에서 나눈 모든 과거 대화 기록을 RAG를 통해 요약 및 검색하여 다음 대화의 컨텍스트로 활용해야 한다. *(Source: Team Member, Project Lead)*
* **FR-3 [사내 데이터 기반 질의응답]:** 시스템은 자동 수집된 사내 데이터(Confluence, Jira, 사내 메신저 등)를 바탕으로 사용자의 질문에 답변해야 한다. *(Source: Team Member)*
* **FR-4 [중복 데이터 병합 요약]:** 회의록이나 기획서 등 버전이 여러 개이거나 중복된 내용이 많은 데이터가 검색될 경우, 최신 맥락에 맞춰 핵심만 요약하여 제공해야 한다. *(Source: Team Member, Project Lead)*
* **FR-5 [맥락 기반 의사결정 추적]:** "저번 달에 A 아키텍처 대신 B를 선택한 이유가 뭐였지?"와 같은 과거 의사결정 맥락 질의에 대해, 대화 히스토리와 사내 데이터를 조합하여 근거를 제시해야 한다. *(Source: Project Lead, Team Member)*

### Non-Functional Requirements (비기능 요구사항, NFR)
* **NFR-1 [토큰 효율성]:** 방대한 장기 대화 세션과 사내 데이터를 처리함에 있어, 내부 RAG의 특성을 활용하여 LLM에 전달되는 프롬프트 토큰 사용량을 최소화해야 한다. *(Source: System Admin)*
* **NFR-2 [정확도 유지]:** 사내 문서에 중복이나 노이즈가 많더라도(예: 동일한 이슈에 대한 여러 작업자의 메모), 환각(Hallucination) 없이 가장 정확한 팩트를 추출해야 한다. *(Source: Team Member)*
* **NFR-3 [동시성 처리]:** 여러 팀원이 동시에 공용 코파일럿 세션에 질의를 하더라도 세션의 맥락이 꼬이지 않고 일관된 상태를 유지해야 한다. *(Source: System Admin, Team Member)*

### Constraints (제약사항)
* **C-1 [블랙박스 RAG 사용]:** 텍스트 검색 및 컨텍스트 압축에는 반드시 사내에 기 구축된 특정 RAG(중복 데이터 처리 및 저토큰 고효율 특성) 모듈만을 거쳐야 한다. *(Source: System Admin)*
* **C-2 [데이터 자동 수집]:** 코파일럿이 참조하는 사내 데이터는 수동 업로드가 아닌, 기존 사내 시스템에서 자동으로 크롤링/수집되는 파이프라인을 거쳐야 한다. *(Source: System Admin, Project Lead)*














# 2. 히스토리 기반 사내 온보딩 및 멘토링 에이전트

## 📌 개요
팀 단위로 수집된 방대한 사내 지식(도메인 지식, 코드베이스, 컨벤션 등)을 바탕으로, 신규 입사자나 전배자가 개별적인 멀티 세션을 가지며 학습과 적응을 진행하는 1:1 맞춤형 멘토링 시스템.

## 👥 Stakeholders (이해관계자)

| 종류 | 역할 | 중요하게 생각하는 품질 속성 (Quality Attributes) |
| :--- | :--- | :--- |
| **Mentee (신규 입사자/전배자)** | 에이전트와 1:1 대화를 통해 사내 지식을 학습하고, 업무 환경 세팅 및 도메인 구조를 파악하는 사용자. | **학습 용이성(Learnability)**, **사용성(Usability)** |
| **Mentor / Team Lead (멘토/리더)** | 멘티의 학습 진행도를 파악하고, 에이전트가 올바른 팀 컨벤션을 안내하고 있는지 확인하는 담당자. | **관측성(Observability)**, **보안성(Security - 권한 관리)** |
| **System Admin (관리자)** | 팀 단위 데이터 수집 파이프라인을 구성하고, 개별 세션의 리소스를 관리하는 담당자. | **확장성(Scalability)**, **격리성(Isolation)** |

---

## 📝 Requirements (요구사항)

### Functional Requirements (기능 요구사항, FR)
* **FR-1 [개별 세션 관리]:** 각 멘티(사용자)별로 독립된 대화 세션이 생성되어야 하며, 이전 날짜의 대화 내용이 다음 날의 학습 맥락으로 이어져야 한다. *(Source: Mentee)*
* **FR-2 [학습 진척도 기반 답변]:** 에이전트는 사용자의 과거 질의응답 히스토리(RAG 요약본)를 분석하여, 사용자가 이미 인지하고 있는 내용은 생략하거나 심화된 내용을 제공하는 등 답변의 수준을 조절해야 한다. *(Source: Mentee, Mentor)*
* **FR-3 [팀 데이터 기반 질의응답]:** 멘티의 질문(예: "우리 팀 UI 컴포넌트 빌드 스크립트는 어떻게 실행하나요?")에 대해, 팀 단위로 수집된 최신 가이드라인과 문서(Confluence 등)를 바탕으로 답변해야 한다. *(Source: Mentee)*
* **FR-4 [주기적 온보딩 리포트]:** 에이전트는 멘티의 누적 대화 세션을 요약하여, 어떤 도메인 지식을 주로 질문했고 어떤 부분에서 막혀있는지 멘토에게 정기적으로 요약 리포트를 제공해야 한다. *(Source: Mentor)*

### Non-Functional Requirements (비기능 요구사항, NFR)
* **NFR-1 [데이터 격리 (Isolation)]:** 대화 세션은 철저히 개인화되어야 하며, 다른 멘티의 질문 히스토리가 현재 멘티의 세션에 영향을 주거나 노출되어서는 안 된다. *(Source: System Admin, Mentor)*
* **NFR-2 [문맥 단절 방지]:** 수주~수개월에 걸친 온보딩 기간 동안 대화가 길어지더라도, RAG를 활용한 세션 요약을 통해 초기 안내 사항을 잊어버리는(Catastrophic Forgetting) 현상을 방지해야 한다. *(Source: Mentee)*
* **NFR-3 [버전 충돌 해소]:** 팀 위키에 오래된 문서와 최신 문서가 혼재(중복)되어 있을 경우, 내부 RAG를 통해 가장 최신의, 혹은 팀에서 공식적으로 채택한 문서를 우선적으로 식별하여 안내해야 한다. *(Source: Mentee, Mentor)*

### Constraints (제약사항)
* **C-1 [블랙박스 RAG 사용]:** 대화 히스토리 요약 및 사내 문서 검색 시 제공된 특정 RAG 구조를 블랙박스 형태로 연동하여 사용해야 한다. *(Source: System Admin)*
* **C-2 [팀 범위 데이터 접근 제한]:** 에이전트가 RAG로 검색하는 사내 데이터의 풀(Pool)은 해당 멘티가 속한 팀의 접근 권한이 허용된 데이터로만 엄격히 제한되어야 한다. *(Source: Mentor, System Admin)*
